# EN_55 — Deployment menu translated (Hire Troops / Equipment / Position / Sortie / Commanders)

Apply `Langrisser1_EN_55_full.xdelta` (md5 159862d001426667fe7ed82da14fa9fe)
to the clean JP `orig.bin`. Supersedes EN_54 (all its content included).
Changed disc LBAs: **394, 403** (LANG1) and **135124–135125, 135147–135149** (FONT.DAT).

## What's new
The deployment menu (the box that opens on your base tile) is now English:
- 兵士配属 → **Hire Troops**
- アイテム装備 → **Equipment**
- 指揮官配置 → **Position**
- 出撃 → **Sortie**
- 指揮官 (right-panel header) → **Commanders**

## Why this was hard (and how it's done)
Those labels are **pre-rendered bold-kanji graphics**, decompressed from IMG.DAT to
video RAM — not text in any table. Rather than crack that compression, the fix rides the
same per-frame hook that already draws the English command ring: the deployment menu's
tilemap is staged in the **same HWRAM buffer (0x060765F8)** and pushed to VRAM through the
**same copy routine** the ring hook intercepts. So the hook now also:
1. detects the deployment menu (two-tile signature: the 兵 and 士 graphic tiles),
2. uploads 92 English letter-tiles (rendered from the Toshiba 8×16 font, half-width
   packed two-per-cell, stroke shade matched to the kanji so the color is identical), and
3. rewrites the menu's tilemap to point at them and blanks the leftover kanji cells.
All five labels fit the existing menu box width, so no layout code was disturbed.

## How EN_54 stayed intact (important)
The English tiles needed storage, and LANG1 was full. To make room **without changing any
EN_54 behavior**, the command ring's 36 glyph tiles were relocated out of LANG1 into free
FONT.DAT space (FONT is resident in VRAM, so the ring's tile DMA just sources from there
now). This freed enough LANG1 room to fold the deployment logic into the existing hook.

Verified byte-for-byte that this didn't change the ring:
- the ring's per-unit write descriptors are **byte-identical** to EN_54;
- the relocated ring tiles are **byte-identical** pixels, landing at the same VRAM cells;
- all ring gate constants unchanged;
- the hook still preserves PR and tail-calls the copy routine exactly as before.
The whole combined hook is 554 bytes (budget 736; the live pointer at 0x64F10 is untouched).

## Revision (color + leftover kanji fix)
First cut rendered the labels dark and left kanji fragments (装備) showing. Root cause: the
menu's pattern-name word uses a **10-bit character field** (plus a flip bit and palette
nibble), so the tile cells must live in charno range 0x1400-0x17FF and each tilemap entry
must carry the same flip/palette bits (0x0400) as the original kanji entries. The first cut
placed tiles at 0x1CF2+ (out of the 10-bit range) with palette/flip bits zeroed, so the
game masked them to the wrong cells (dark/garbled) and the kanji stayed. Fixed: tiles now
live at charno 0x1754-0x17AF, entries = 0x0400 | (charno-0x1400) — matching the kanji entry
format exactly, so the letters render the same **white** as the kanji did, and every
original kanji cell is cleared. md5 now 159862d001426667fe7ed82da14fa9fe.



## Revision 2 (grey -> white, and remaining kanji cleared)
The corrected-charno build still rendered **grey** and left one floating kanji. Two fixes:
- **Colour:** the menu palette has index **1 = white**, index **3 = grey** (confirmed against
  the live screenshot: "Ledin" white = index 1; my labels sampled at index 3 = grey). The
  save-state CRAM byte-order was misleading. Letters now use **stroke nibble 1**, so they render
  the same white as the surrounding UI.
- **Leftover kanji:** the JP labels' kanji/filler span cols **4-15** (the last kanji of
  アイテム装備 sits at col 15). The earlier clear stopped at col 14, leaving col 15's kanji.
  Clears now cover cols **4-15** (left rows 6-13) and **21-26** (header) fully; every
  label-region cell is written (verified 0 uncovered). Staging writes grew 110 -> 116 and the
  hook's loop count was made data-driven so it can't drift again.
md5 now 159862d001426667fe7ed82da14fa9fe.



## Revision 3 (sub-screen title bars)
Selecting Hire Troops / Equipment / Position opens a sub-screen whose **title bar** still
showed the Japanese (兵士配属 / アイテム装備 / 指揮官配置). These title bars use the *same*
staging buffer (0x060765F8) and the *same* font tiles as the main menu, just placed at
**row 2** instead of the menu rows. The hook now dispatches on the staging buffer: if
buf[0x308] (row6 col4) is the main-menu signature it applies the main table; else if
buf[0x108] (row2 col4) is a title key (0x043C Hire / 0x0400 Equip / 0x0428 Position) it
applies that title's table. All four tables live in one FONT.DAT blob; the English tiles are
already resident (shared with the main menu), so no new tiles were needed. Sortie has no
sub-screen and Commanders is a header, so only three titles are handled. md5 now
159862d001426667fe7ed82da14fa9fe.
